# xiaodongq 博客写作风格指南

本skill用于按照xiaodongq的博客风格撰写文章。

特别注意：保持真实，不要虚构我进行过xxx，如果需要可以进行确认。如果是AI自动写文章，就说了解/调研过xxx，不要说我试过xxx。

## 1. Front Matter 格式

每篇文章必须包含以下Front Matter：

```yaml
---
title: 文章标题
description: 文章描述
categories: [分类1, 分类2]
tags: [标签1, 标签2]
---
```

**注意事项：**
- 不需要 `date` 字段（日期通过文件名体现）
- 如需Mermaid图表，添加 `mermaid: true`
- categories和tags使用数组格式
- description 要简洁，一句话说明核心价值

## 2. 标题格式

### 2.1. 文章标题格式

技术类系列文章使用统一前缀格式：
- `AI能力集 -- xxx` - AI能力集系列
- `[类别] xxx` - 其他分类文章

文章标题要简洁清晰，直击痛点，能一句话说明价值。比如：
- ✅ "基于 Obsidian + LLM Wiki 开始构建个人知识工程"
- ❌ "如何优雅地使用 Obsidian 和 LLM 构建个人知识库的完整指南"

### 2.2. 多级目录结构

使用以下层级结构：

```
## 1. 引言        # 一级标题：章节号 + 标题
### 1.1. xxx       # 二级标题：小节号 + 标题
#### 1.1.1. xxx    # 三级标题（必要时使用）
```

**示例：**

```
## 1. 引言

## 2. MiniMax MCP

### 2.1. 安装和使用

### 2.2. MCP实现流程

#### 2.2.1. 核心模块
```

## 3. 语言风格

### 3.1. 混合格式
- 中文为主，关键技术术语保留英文
- 不要太口语化

### 3.2. 记录AI交互过程
文章中经常包含与AI的实际对话/交互记录：
```markdown
XX帮我安装xxx

⏺ Bash(命令)
  ⎿  输出结果
```

### 3.3. 引言写法
引言是文章的门面，决定读者会不会继续看下去。要做到：
- **一句话点明价值**：读者为什么要关心这个？
- **痛点前置**：先说传统方案的缺点，再说新方案的好
- **保持真实**：不要虚构我进行过xxx，如果需要可以进行确认

❌ 避免：泛泛而谈的背景介绍、与主题无关的絮叨
✅ 推荐：问题驱动、需求驱动、有感而发

## 4. 排版规范

### 4.1. 代码块
**必须指定语言：**
```sh
```python
```yaml
```json
```html
```

**代码块内注释：**
- 使用 `#` 注释
- 命令行输出保持原始格式

### 4.2. 表格格式
```markdown
| 列1 | 列2 | 列3 |
|-----|-----|-----|
| 内容 | 内容 | 内容 |
```

### 4.3. Emoji使用
在适当场景使用emoji增加可读性：
- ✅ 完成状态
- ❌ 失败/问题
- 📋 列表/清单
- ⏺ 执行中/进度
- 🚀 启动/完成
- 💡 提示/要点

### 4.4. 分隔线
章节之间使用分隔线：
```markdown
---
```

### 4.5. 图片
```markdown
![图片描述](/images/图片路径)
```

### 4.6. 实用要点
- **相关链接放引言**：参考链接、仓库链接统一放引言，不在正文中零散分布
- **表格代替列表**：多个插件对比用表格，比列表更清晰
- **截图要适度**：关键效果展示放截图，但不要滥用
- **分隔线不要太多**：每个大章节结束后放一条即可，避免过度分割

## 6. 问题记录格式

如有踩坑记录，使用专门章节：
```markdown
## N. 问题记录

### N.1. 问题1

错误信息：
```
错误内容
```

解决方式：
...
```

## 7. 写作要点

### 7.1. 实用性优先
- 记录真实踩坑和解决方案
- 包含可运行的命令和代码
- 问题解决过程要完整

### 7.2. 对比分析
- 适当使用表格对比不同方案
- 包含优缺点分析

### 7.3. 实战感受
- **使用体验很重要**：记录实际用下来的感受，包括踩坑、体会和"银弹特性"
- **真实优于完美**：不满意的实验结果也要记录，让读者少走弯路
- **个人思考要有**：不只是功能清单，要写出"为什么用这个方案"
- **适度反思**：对方案局限性的诚实讨论

### 7.4. 引用资源
- 引用相关文档、仓库、公众号文章
- 使用完整URL

### 7.5. 实战记录
- 包含实际执行命令和输出
- 记录AI交互过程
- 保留真实结果（包括不满意的结果）

## 8. 系列文章关联

### 8.1. 引用同系列文章
```markdown
[之前](https://xiaodongq.github.io/日期/文章slug/) 相关内容
```

### 8.2. GitHub仓库链接
```markdown
相关仓库：[仓库名](https://github.com/xxx/xxx)
```

## 9. 总结章节写法

好的总结要"兜得住"，让读者记得住核心价值：
- **一句话概括核心价值**：这套方案解决了什么问题
- **点出飞轮效应**：长期积累后的收益（如"每加一个源，Wiki就更丰富"）
- **留个钩子**：暗示后续可能的探索方向

❌ 避免：简单重复功能列表、流水账式回顾
✅ 推荐：有感而发、有洞见、有回味

## 10. 写文章流程

1. **确定主题和系列**：判断属于哪个系列
2. **确定Front Matter**：title, categories, tags
3. **编写引言**：背景、需求、相关链接
4. **按逻辑结构编写正文**：概念→安装→使用→示例→问题
5. **添加代码块和表格**：确保代码有语言标识
6. **补充问题记录**：如有踩坑，单独章节记录
7. **检查格式**：emoji、分隔线、图片路径
